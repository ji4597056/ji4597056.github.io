建立递归思想

技术
  • 算法

start

  最近一段时间,开始一直做算法题,虽说这个过程很虐心,但是痛并快乐着,我并不会因为题目写不出来就死磕,我相信这是个学习的过程,不会写就学习算法思想,不要失去信心.但是很多题目需要用递归方法可以优雅地解决问题实在让我不得法门.虽然也许我能看懂算法实现,但我始终觉得自己是写不出这样的代码,就算再次遇到同类型的问题,我也无法做到用递归优雅地解决问题.(这里姑且不论递归的性能和可能引起的栈溢出问题,递归本身是一种清晰优雅的实现)

  每每看到优雅的递归解决方案就很失落,为何我就是不能写出复杂地递归算法呢?归根结底应该是算法思想的问题.毕竟循环迭代的思想先入为主.所以决定死磕递归思想,反复地去写递归算法题,直到建立这种思想.

content

要理解递归,先理解递归

  递归简而言之就是函数自己调用自己,这个层层不断调用的过程称之为递,当然,如果放任函数无穷无尽地自己调用自己岂不是死循环,所以必须得有一个终结条件,当函数满足了这个终结条件便不会再自己调用自己.此时,会将返回结果层层不断地响应给调用者,这个过程称之为归,直到响应到达最初的调用者,便得到了最终结果.

  递归思想本身而言,并不算高深,可以用很直观的语言表达.但是往往在使用递归思想解决问题时,就变得无从下手,不得其法.究其原因,可能是我们解决问题的思维有问题.我们需要在思想上转变,不要试图用常规的思考方式,想着一步到位解决问题.此时,我们需要变得鼠目寸光一点,尝试去递推问题,缩减问题的规模,缩减到最后,就找到了问题的特例形式,这时可以很轻易返回一个确切的结果.这个确切的结果在归的过程中会不断’壮大’,直到归无可归,这是也就得到了问题的最终结果.

道理我都懂,但是我不会

  所谓思想,只可意会不可言传.很多大师大神都总结过递归思想,他们的解释肯定比我更专业更精炼.但是当你并没有真正对这种思想有’感觉’时,似乎看再专业的解释也是朦朦胧胧.当然,我也经常有这种感觉,会觉得自己无法理解思想的精髓,是不是自己笨呢?也许确实自己并不算聪明,但是幸运的是,绝大部分人其实聪明的不明显.此时,多安慰安慰自己,不要失去自信,算法大道,路漫漫其修远兮.

  既然做不到瞬间领悟递归思想,那么怎样锻炼这种递归的思维模式呢?当然是做题、做题再做题啦!推荐一本书,也是很多大神都推荐的一本奇书–计算机程序的构造与解释.用心钻研,一定会有收获.

  这里总结下大概的递归问题解题思路.

  1. 实现问题最基本情况的解法.
  2. 将复杂的问题划分成更小规模的子问题.
  3. 得到子问题返回的结果,组合成所求结果.

验证

  这里需要多提的一点是,递归和我们学数学时的数学归纳法极为相似.数学归纳法不多介绍,这里提到数学归纳法主要为了表达两点.

  1. 当我们用递归思想实现一个复杂问题的解决方案后,不妨先验证特例情况是否正确,再假设情景为N时结果成立,此时再验证情景为N+1时依然成立,则解决方案正确.
  2. 当面对一个复杂问题束手无策时,我们先假设已有一个解决场景为N时的解决方案,此时再思考当情景规模变为N+1时,我们该如何利用解决情景N的方案去解决情景N+1的规模.

刷题

  理论就简单写那么多吧,思想这东西只可意会,还是刷题实在.Let’s GO!

链表反转

  1. 缩小问题规模为head为null或者链表只有1个节点,直接返回head.
  2. 考虑此时已有reverse(head.next)的解决方案,想要加上head,只需head.next.next=head,head.next=null.
  3. 返回newHead,即reverse(head.next).

快速排序

  1. 缩小问题规模为快速排序数组中low >= high的元素(即只有一个元素),则直接return.
  2. 找到切分点j.
  3. 排序数组[low, j-1].
  4. 排序数组[j+1, high].

归并排序

  1. 缩小问题规模为归并排序数组中low >= high的元素(即只有一个元素),则直接return.
  2. 找到中值点mid.
  3. 归并排序[low, mid].
  4. 归并排序[mid+1, high].
  5. 合并[low, high]

二叉树遍历

  1. 以前序遍历为例,若当前节点为null,则直接return.
  2. 打印当前节点
  3. 遍历左子树
  4. 遍历右子数

合并多个有序链表

  1. 根据二分法思想,当low >= high时,则直接返回Lists[low]
  2. 找到中值点mid
  3. 排序[low, mid]
  4. 排序[mid+1, high]
  5. 合并(这里也是递归调用自身)

  刷题就刷到这里吧,这几题比较典型,也不算难,核心思想大致都一样,思考问题规模最小时的解决方案,再假设规模为N时的解决方案可行,根据解决规模为N+1的方案从而得出最终方案.

end

  最近沉迷算法,一直被递归肆虐,这时候就需要展示男人的魄力,没有思想,就创造思想,刷个上百题,一切问题就不成问题.>_<!
